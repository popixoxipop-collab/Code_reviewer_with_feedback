# [Methodology Research] 인지-판단-피드백 3블록 파이프라인 — 실증 검증 결과 및 결론

## 1. 이 리서치의 위치

B2B 개발자 사고력 검증 플랫폼 설계 리서치(외부 문서)는 MVP 전체(데이터 파이프라인 + 인터뷰 엔진 + 채점 엔진)를
다룬다. 이 저장소에서 실제로 코드를 짜고 돌려서 검증한 범위는 이 중 **①데이터 파이프라인(인지)** 과
**③평가 엔진의 "신호 신뢰도 관리"(판단)** 두 계층이며, **②인터뷰 엔진(Socratic Red-Teaming 동적 루프)은
구현하지 않았음**을 먼저 명확히 한다.

| 외부 문서 | 이 저장소 실증 대상 | 상태 |
|---|---|---|
| §2.1 데이터 파이프라인 | 인지 블록(Tier A 구조 스캔 + Tier B 위험 트리거 스캔) | ✅ 실증 완료 |
| §2.3 평가 엔진(신호→점수) | 판단 블록(규칙 기반 채점 + 관용 패턴 재귀 필터) | ✅ 실증 완료 |
| §2.2 인터뷰 프롬프트(Socratic Red-Teaming) | 피드백 블록(Depth Ladder 7단계) | ⚠️ 정적 템플릿만 존재, 동적 멀티턴 반례 생성 루프 미구현 |

## 2. 실증 비교: 데이터 파이프라인 (§2.1 대응)

| 검토 방법론 | 외부 문서 판정 | 실증 결과 |
|---|---|---|
| A. Graph RAG (AST 기반) | 탈락(2개월 내 TS/Java 양방향 파서 구축 불가) | **부분 반박**: AST 없이 정규식 기반 import 스캔만으로 fan-in 그래프 구축이 됨(공개 repo Study-Match-, 소스 10파일 전체 스캔). "AST 파서 구축"과 "그래프 신호 확보"는 분리 가능한 문제였음 |
| B. Agentic Browsing | 탈락(비용/지연) | 미검증(애초에 채택하지 않음, 반박할 근거 없음) |
| C. Heuristic 추출(커밋 빈도) | **최종 채택** | 커밋 빈도 대신 **fan-in(구조) + 위험 키워드(내용)** 2-tier를 사용. 실증에서 fan-in 최고 파일(firebase.ts)과 실제 위험 지점(내용 기반)이 **서로 다른 파일**이었음 — 단일 휴리스틱(커밋 빈도든 fan-in이든)만으로는 반드시 사각지대가 생긴다는 게 실측으로 드러남 |

**교차검증 발견**: "AST 기반은 리소스 초과"라는 판정 자체는 타당하나, 그 대안이 꼭 "커밋 빈도"일 필요는 없다.
Tier A(정규식 import 스캔)는 AST보다 구현 비용이 낮으면서도 커밋 빈도가 못 잡는 신호(구조적 고립 —
다른 화면은 전부 Firestore를 쓰는데 한 화면만 하드코딩 데이터)를 잡아냄. 여력이 되면 "커밋 빈도 +
import 기반 fan-in/고립도" 2-신호 결합을 검토할 가치가 있음.

## 3. 실증 비교: 신호 신뢰도 관리 (§2.3 평가 엔진과 연결되는 지점)

키워드 매칭 기반 채점의 실패조합("키워드만 맞으면 만점")과 정확히 같은 클래스의 실패가 판단 블록에서
실측으로 재현됐다.

### 실패 조합 (필터 없음): [구조 신호(fan-in)] × [단순 임계값 판정]
- 판정 로직: "fan-in 6 이상 → 질문가치 상"
- 실제 산출: `App.tsx`의 `useAuth` Context(fan-in=6)가 질문가치 "상"으로 확정, 우선순위 리포트에 노출
- **평가**: React 컴포넌트 6개 규모에서 Context는 프레임워크 관용 패턴에 가까운데, 구조 지표만 보면
  "설계 판단"과 구분이 안 됨 — 신호가 구조든 키워드든, 출처를 검증 안 하면 곧이곧대로 속는다.

### 성공 조합(채택안): [구조+위험 신호] × [규칙 채점 + 관용 패턴 재귀 필터]
- 절차: idiom_hook에 "그냥 관용 패턴이었다"는 피드백을 3회 누적 → threshold 도달 시 candidate→confirmed
  자동 승격 → 이후 판단 실행 시 자동 적용
- 실제 산출: `App.tsx` 질문가치 **상→하 자동 강등**, 우선순위에 강등 사유 자동 기록. 동시에
  `Competitions.tsx` 데이터 단절(진짜 검증 가치 있는 신호)은 필터에 안 걸리고 최우선 유지
- **평가**: 신호를 곧이곧대로 안 믿고 근거 기반 필터를 거쳐야 한다는 결론에 비-LLM, 규칙 기반,
  threshold 재귀 승격 방식으로 도달함. 단일 피드백으로 즉시 확정하지 않고 3회 반복 확인 후 승격시킨 것도
  성급한 자동판정이 시스템을 무력화하는 것을 막기 위함(RunPod_Deploy_Agent BLOCK 남발 금지 사례,
  arXiv:2603.18059 참고).

## 4. 정직하게 보고하는 미검증 영역

| 항목 | 상태 |
|---|---|
| Socratic Red-Teaming 동적 반례 생성 | ❌ 미구현 — Depth Ladder는 사람이 수동으로 채운 정적 7단계 질문지 |
| LLM-as-a-Judge Rubric 채점 | ❌ 미구현 — 판단 블록은 규칙 기반(상/중/하)이지 1~5점 정량 채점기가 아님 |
| Evidence-based 팩트체크 | ❌ 미구현 |
| 다국어(Java 포함) 관용 패턴 필터 | ⚠️ 구조만 존재, 실제 시드 데이터 없음(javascript만 실증됨) |

즉 검증한 건 "타겟을 어떻게 찾고 어떤 신호를 신뢰할지"이지, 압박 질문을 실시간으로 어떻게 만드는가는
아직 손대지 않았다.

## 5. 결론 및 통합 제안

1. **데이터 정제(Heuristic Extraction)**: 커밋 빈도 단일 신호를 그대로 채택하되, 여력이 되면
   "구조적 fan-in/고립도"를 2차 신호로 결합 검토 — 두 신호가 서로 다른 파일을 가리켰다는 근거 있음
2. **검증 엔진(Socratic Red-Teaming)**: 이 저장소 스택으로 아직 검증 안 된 영역. Socratic × LLM-Judge
   조합이 맞는 방향이며, 판단 블록은 이 엔진의 **전처리 단계**로 통합하는 게 자연스러움(D8)
3. **평가 엔진(LLM-as-a-Judge)**: idiom 필터를 LLM-Judge 완전 대체가 아니라 **앞단 노이즈 제거기**로
   배치 제안 — judge 프롬프트에 들어가는 finding 자체의 관용패턴 노이즈를 줄여 judge의 일관성/비용을
   개선할 수 있음(가설, 미검증)

## 6. 설계 결정 로그

- **D8** — 인지/판단 블록을 LLM-Judge 앞단의 "전처리 필터"로 위치시킴 (대체 아님)
  - WHY: LLM-as-a-Judge는 자연어 논증 품질(트레이드오프 인지, 반례 수용) 평가에 강하고, 규칙 기반
    필터는 "구조적 신호의 신뢰도" 판정에 강함 — 역할이 다르므로 경쟁이 아니라 파이프라인 전/후 배치가 맞음
  - COST: 파이프라인 단계가 하나 늘어 레이턴시가 소폭 증가할 수 있음
  - EXIT: 필터링된 결과와 raw finding을 LLM-Judge에 각각 넣어봐서 품질 차이가 없다고 실측되면,
    `judgment/score_findings.py` 호출을 건너뛰고 raw finding을 그대로 넘기는 구성으로 되돌릴 수 있음
    (코드 삭제 불필요, 호출만 스킵)
- **D9** — Socratic Red-Teaming(동적 반례 생성) 엔진은 이 리서치 범위에서 검증하지 않았음을 명시
  - WHY: 실측한 건 "타겟팅과 신호 신뢰도"이지 "실시간 압박 질문 생성"이 아님 — 별도 검증 없이
    성공조합이라 단정하면 근거 없는 낙관이 됨
  - COST: 이 리서치만으로는 Socratic × LLM-Judge 성공조합이 이 스택 위에서도 재현되는지 보장 못함
  - EXIT: [`feedback/depth_ladder_template.md`](./feedback/depth_ladder_template.md)를 실시간 LLM
    호출로 자동화([`README.md`](./README.md) "다음 단계" 3번)해서 실제 멀티턴 시뮬레이션을 돌려봐야
    결론 가능
