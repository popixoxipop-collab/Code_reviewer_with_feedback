What D94 Actually Changed

D93까지는 strong / weak / improving 답변이 전부 고정 텍스트였습니다. D94는 turn_engine.run_decision_point()의 answer_fn 시임에 claude -p --model sonnet --safe-mode를 연결해, 각 턴에서 학생 답변을 Sonnet이 실시간으로 생성하도록 바꿨습니다.

  • 모델 후보군은 NVIDIA Build 카탈로그를 다시 조회해 16개로 확장했습니다.
  • 실패분은 전역 30rpm 슬라이딩 윈도우 게이팅으로 재시도해 43건을 회복했습니다.
  • 정규식 판정은 reflection 문구 다양성에 취약했지만, LLM-as-judge는 자기수정 축을 안정적으로 포착했습니다.

Interpretation

신호가 분명한 부분: step-3.5-flash, qwen3-next-80b, mistral-medium-3.5, llama-4-maverick는 strong > weak 분리를 안정적으로 만들었습니다.

주의할 부분: D94는 실행 환경과 레이트리밋 민감도가 높아서, 낮은 성공률 모델은 품질 해석보다 인프라 원인 진단이 우선입니다.

중요: 아래 두 문제 모델은 "고장난 모델"이 아니라 각기 다른 실행 조건 문제로 판명됐습니다.

Blocked Model 01

meta/llama-3.3-70b-instruct

raw HTTP 진단에서 NVIDIA가 직접 503 ResourceExhausted: Worker local total request limit reached (153/16)를 반환했습니다. 즉 모델 자체 결함이 아니라 워커 큐 과부하였습니다.

  • 진단 단일 호출 1회차: 124.7초 후 503
  • 진단 단일 호출 2회차: 188초 후 정상 응답
  • 의미: 기존 timeout_s=120는 짧았고, 시간대와 병렬도도 영향을 줍니다
  • 다음 조정점: timeout 200초+, 병렬도 축소, 다른 시간대 단독 재시도
Blocked Model 02

nvidia/llama-3.3-nemotron-super-49b-v1.5

reasoning 모델 특성상 내부 reasoning_content가 길고, max_tokens=512로는 chain-of-thought 단계에서 finish_reason="length"가 나며 content:null로 끝났습니다.

  • 진단 3회 모두 같은 현상 재현
  • 문제는 서버 장애보다 tool-calling 전 단계의 토큰 부족
  • 다음 조정점: turn_engine.generate_question(..., max_tokens=2048)
  • 보조 조정점: 병렬도 축소로 워커 큐 압박 감소
