## 0. 용어 고정 (혼동 방지 — 문서 전체에서 이 표기를 강제)

이 repo에는 "hook"이라는 이름이 이미 두 가지 완전히 다른 목적으로 존재하게 된다. 계획서·코드·커밋 메시지에서 아래 표기를 고정한다.

| 표기 | 실체 | 목적 | 대상 |
|---|---|---|---|
| **루브릭 보정 훅** | `judgment/idiom_hook.py` · `isolation_hook.py` · `tier_b_hook.py` · `subrubric_hook.py` | 채점 루브릭의 **오탐(false positive)을 팀원 피드백으로 재귀 억제** — 3인 판정 시 candidate→confirmed 승격, 질문가치 자동 강등 | 시스템 내부 (판단 블록의 자기 교정) |
| *(참고)* `feedback/reflection_hook.py` | 이름은 비슷하지만 위 "루브릭 보정 훅"과 무관 — **파이프라인 03** turn_engine의 반성/자기교정 신호 탐지 파일. `judgment/`가 아니라 `feedback/`에 위치. 혼동 방지를 위해 이 문서에서는 별도 취급하며 §5 Hook File 설계에도 관여하지 않는다. | — |
| **Hook File** | 신규 개념 (`hookfile/` 블록으로 신설 예정) | 교육생 개인의 **측정·질의응답·시행착오 누적 결과를 수료 시점에 남기는 개인화 처방 자산** (RPT-02 산출물). CUR-02(교안 매핑)·ENG-01(취약축 진단)·DASH-15(기술영역 분류)·MEAS-06(소크라테스 질의응답 결과)을 하나의 실행 가능한 포맷으로 압축 | 교육생 개인 (수료 후 개발 환경에 장착) |

- 둘은 **주체도 방향도 다르다**: 루브릭 보정 훅은 "시스템이 덜 틀리게" 만들고, Hook File은 "학생이 같은 시행착오를 반복하지 않게" 만든다.
- Hook File의 그릇(전달 포맷)은 Claude Code CLAUDE.md / Cursor Rules / AGENTS.md / Claude Code Hooks(JSON 이벤트 핸들러)와 같은 계열이지만, **내용물은 사용자가 임의로 쓴 정적 규칙이 아니라 본인의 코드 진단·질의응답·FR-04-01 5축 평가·교안 페이지 매핑에서 나온 근거 기반 압축**이다 — 이 프레이밍은 사용자가 이미 확정했고 재논쟁 대상이 아니다.
- **측정 중 처방 금지 원칙(확정)**: 측정 기간 중 Hook File을 주입하면 ENG-02/INTV-06/DASH-07 측정 항목의 인과가 오염된다. Hook File은 최종 회차 완료 후 RPT-02 산출물로만 제공한다. 이 계획의 실험 설계(5절)는 이 원칙을 침해하지 않고 전/후 비교를 가능하게 만드는 것이 핵심이다.

## 1. 현재 상태 — 세 파이프라인이 어디에 서 있는가

세 파이프라인은 같은 팀 스펙에서 출발했지만 코드 상 import/데이터 의존이 없는 독립 파이프라인이다.

### 1.1 파이프라인 01 — 교안 분석 (Java Curriculum → NVIDIA Build)

- 파일: `scripts/java_curriculum_nvidia_pipeline.py`(메인), `scripts/nvidia_keypool_traffic_test.py`
- 실제 단계(코드 확인): PDF(251p) → `pdftotext` 10p 청크(26개) → 청크별 병렬 NVIDIA 호출(`analyse_chunk`, 키 로테이션) → `make_unit_map`(모든 unit/concept/code/caution에 `source_pages` 보존) → `refine_once` 루프(페이지 커버리지·중복 개념·unit 경계·graph readiness 자기감사) → `build_graph`(graphify 호환) → `generate_questions`(**모든 질문이 `source_node_ids`+`source_pages` 인용 강제**, rationale/good_answer_signal/common_gap 필드 포함)
- 상태: 3청크 스모크는 성공(D95), 251p 풀런은 NVIDIA 서빙 지연/JSON 안정성에 막혀 실험 단계. **정량 벤치마크 0건 — 이번 계획의 핵심 갭 1.**

### 1.2 파이프라인 02 — 코드 분석 (Cognition + Judgment, 정적분석)

- 파일: `cognition/two_tier_scan.py`(2계층 스캔, cost_saved_ratio) · `judgment/subrubric.py`(규칙기반 4서브축×3축, 0~12 총점→상/중/하) · `judgment/score_findings.py`(메인) · 루브릭 보정 훅 3종 · `pipeline/run_pipeline.py`(오케스트레이션, injections before/after diff) · `followup_generator.py` · `escalation_hook.py`
- 구조적 사실(설계에 결정적): **주 경로 전체가 결정론(regex/규칙)이고 LLM 콜이 0건**이다. 대신 상태 파일(`judgment/idioms/<lang>/idiom_patterns.json` 등)에 의존해 "같은 코드라도 상태가 다르면 결과가 달라진다"(D5 COST에 이미 문서화).
- 팀 맥락: MEAS-02(Decision Point 추출)는 이 정적분석이 아니라 **순수 LLM으로 대체 확정(Locked)** — 이 파이프라인은 5개 경쟁 방법론 중 하나로서의 기록(D1~D53)이다. 벤치마크의 목적은 팀 결정 재론이 아니라 **방법론 비교표의 공정한 정량 완결**이다.
- 상태: 기능 구현 완료 + 41개 외부 repo(D74/D75)·`examples/` 7세트 실증. **파이프라인 03 수준의 4축 정량 벤치마크는 0건 — 핵심 갭 2.**

### 1.3 파이프라인 03 — 질의응답 (turn_engine) — 완료, 참조 템플릿

## 2. D118 (제안) — 3파이프라인 벤치마크 등가화의 공통 원칙

축 이름은 유지하되 **정의는 각 파이프라인의 실제 최종 산출물에 맞게 재정의**한다. "3처럼 하라"를 문자 그대로 이식하면 P02(결정론·LLM 콜 0)에서는 재현성이 무의미하게 1.0이 되고, P01(장문 JSON 추출)에서는 완전일치율이 무의미하게 0이 된다.

- WHY: 4축의 **비교 가능성은 산식 계보에서 온다** — (1) 안정성은 항상 "스펙이 요구하는 전체 역할의 end-to-end 곱"(D108c), (2) 재현성 판정은 "정확히 REPEATS개 성공한 입력만 완전일치 판정"(D115의 집계 함정 — ≥2회 완화가 커버리지 낮은 쪽을 역설적으로 부풀리는 편향을 실측 반증으로 확인) + mode_agreement/coverage를 진단용 보조 지표로 병기, (3) 실패도 데이터(무재시도 설계), (4) 인프라 원인은 점수 면죄부가 아니라 annotation(D109/D111/D116의 `SERVING_OUTAGE`/`VOLUME_QUOTA_BURST_INCIDENT` 계보).
- 추가 공통 규약:
  - **REPEATS가 다른 identical_rate끼리 비교 금지** — identical_rate는 REPEATS에 단조 민감(D115). 파이프라인별 anchor REPEATS를 명시하고 summary JSON에 `reproducibility_repeats`를 항상 기록한다(P03과 동일 필드명).
  - **페이싱**: D112/D113이 증명한 "60rpm 안전"은 step-3.5-flash 단일 모델 검증이며 전 모델 일반화 불가(D116 — glm/deepseek도 시간 단위 총량 쿼타 버킷 보유 확인). P01 벤치는 qwen 단일 모델 장시간 실행이므로 **보수적 기본(12~20rpm, workers≤4)에서 시작**하고 429 관측 시 즉시 하향.
  - timeout/max_tokens는 `timeout_config.py` 중앙 상수만 사용(D98/D104 hook이 강제).
  - 결과 공개 3면(README, `docs/index.html`, `docs/pipelines.html`) 동기화를 각 마일스톤 종료 조건에 포함(D100 재발 방지).
- COST: 축 재정의로 인해 "P01의 재현성 0.6"과 "P03의 재현성 0.6"은 같은 숫자라도 같은 의미가 아니다 — 각 summary에 축 정의 문자열(`note` 필드)을 내장해 표만 떠돌아도 오독되지 않게 한다.
- EXIT: 재정의가 과했다고 판명되면(예: P01 구조 시그니처가 너무 관대) 판정 함수만 교체하면 되도록 원자료(raw 반복 출력 전문)를 P03과 동일하게 전량 보존한다 — `--aggregate-only` 재집계 패턴 재사용.

## 3. (A-1) D119 (제안) — 파이프라인 02 4축 벤치마크: 결정론 파이프라인 적응

최종 산출물 = `score_findings.py`의 판단(findings + 3축 서브루브릭 총점 + 상/중/하 버킷). LLM 콜이 없으므로 **API 비용 0, 전량 로컬 CPU**. 가장 싸고 빠른 마일스톤이라 첫 순서로 배치한다.

### 3.1 축 재정의

| 축 | P03 정의 | P02 재정의 | 측정 방법 |
|---|---|---|---|
| 안정성 | 질문생성×채점 콜 성공률 | **스캔 성공률 × 판단 성공률의 end-to-end 곱** (crash-free + 출력 스키마 유효 + 필수 필드 non-null) | `examples/` 7세트 + D74/D75 외부 41 repo 전수 실행. 진입점 다중화 monorepo(Shadowbroker에서 확인된 미해결 한계 #13) 같은 알려진 실패 모드가 성공률에 정직하게 반영됨 |
| 정밀도 | candidate-as-grader strong>weak 분리 | **사람 확정 라벨 기준 "진짜 결정 지점" vs "오탐/관용 패턴"의 서브루브릭 총점 분리도** (strong>weak의 정확한 대응물) + 보조로 meas02 방식의 reference-set coverage(D66 명명 준수 — "recall"이라 부르지 않음) | 기존 확정 라벨(루브릭 보정 훅 피드백 로그, D39~41 Codex 판정 6건, Shadowbroker tier_b ledger)은 소수라 **라벨링 스프린트 필요**: corpus에서 층화 표본 ~50 findings, 팀원 2인 독립 라벨(질문가치 있음/없음), 라벨러 간 일치도(kappa) 보고. README 미해결 #14("사람 채점과 비교하는 검증은 여전히 안 함")를 이 항목이 정면으로 해소 |
| 재현성 | temp=0 100회 완전일치율 | 3개 층으로 분해: **(a) 고정 상태×동일 입력 100회 반복 — 정규화 JSON 바이트 완전일치율**(결정론 검증: 기대값 1.0, 미달 = os.walk/셋 순회 순서류 버그 발견), **(b) 상태 민감도 — 상태 스냅샷 3종(빈 상태/현재 confirmed/시드 완충)에서 같은 입력의 버킷 변화율**(버그가 아니라 D5 COST가 문서화한 의도된 상태 의존 — 분리 보고), (c) 선택: Python 버전 2종 교차 | (a)(b)는 로컬 CPU로 수 분. REPEATS=100은 **비용이 0이라서** P03 headline과 동일 앵커를 그대로 쓴다(가장 싼 엄밀성) |
| 속도 | 평균 응답시간 | **repo당 wall-clock**(스캔/판단 분리 계측) + KLOC당 처리 시간, 그리고 이 파이프라인 고유 지표인 `cost_saved_ratio`를 병기 | 로컬 계측 |

- 종합지수: P02는 모델 후보가 없어 min-max 대상이 없다 — **절대값 4축 스코어카드**로 보고하고, 종합 랭킹이 필요한 비교축은 아래 확정 트랙에서만 생성.

### 4.1 축 재정의

| 축 | P02와 달리 P01에서의 정의 | 측정 방법 |
|---|---|---|
| 안정성 | **청크 분석 성공률 × refine 성공률 × 질문생성 성공률** (3스테이지 end-to-end 곱, D108c 계보). 성공 = 유효 JSON + 필수 필드 + `source_pages` 규칙 위반 없음 | 풀런(26청크) 3회 반복의 스테이지별 성공률 + 재현성 측정분의 모든 콜을 안정성 표본으로 재사용(P03이 main 하니스와 regrade를 겸용한 방식). 실패 원인은 인프라/모델 결함 annotation으로 구분(D109 계보) — D95가 관측한 "서빙 지연/JSON 불안정"이 처음으로 수치가 됨 |
| 정밀도 | **출처 정합률(provenance precision)** — 이 파이프라인의 설계 목표가 환각·출처누락 방지이므로 strong>weak의 대응물은 "인용한 페이지에 그 내용이 실제로 있는가"다. 2단 검증: (1단) 결정론 텍스트 대조 — concept 라벨/summary의 핵심 토큰이 인용 페이지의 pdftotext 원문에 존재하는지(콜 0건, 보수적·저재현율), (2단) 1단 불일치분만 별도 판정 — 사람 표본 감사(~30건) 또는 **생성 모델과 다른 모델**의 judge 콜(자기 확증 방지) | 보조 지표: 질문의 `source_node_ids`가 실제 graph 노드를 가리키는 비율(기계 검증), 페이지 커버리지(unit_map이 대표하는 페이지 비율 — refine 감사 로직 재사용). **1단 자동 검증기 자체의 신뢰도를 사람 표본과 대조해 함께 보고**(Data-First: 검증기도 검증) |
| 재현성 | temp=0 반복 시 **구조 시그니처 완전일치율**. 장문 JSON의 raw 바이트 일치는 사실상 0이 되므로, 판정 벡터를 정규화한다: 청크→ (unit 경계, 정렬된 concept 라벨 집합, source_pages 집합), 질문생성→ (질문 수, 정렬된 (unit, source_pages) 시그니처 집합). 보조: concept 집합 Jaccard 평균, `reproducibility_mode_agreement_rate`(P03 D115 신설 지표 재사용) | **층화 2계층**: (i) 청크 층 — 26청크 중 대표 5청크(문서 앞/중/뒤 + 코드예제 밀집 + 텍스트 밀집)를 고정하고 각 REPEATS회, (ii) 질문생성 층 — 고정 graph 입력 1건에 REPEATS회. 시그니처 정규화 규칙은 파일럿에서 확정(아래 4.2) |

### 4.2 REPEATS 산정 — pilot-then-size (Data-First)

"100회가 항상 정답"이 아니라 **판별하려는 차이 대비 신뢰구간과 콜 비용으로 결정**한다. identical_rate는 이항 비율이므로 95% CI 반폭 ≈ 1.96·√(p(1-p)/n), 최악 p=0.5 기준:

| REPEATS | CI 반폭(최악) | 청크 층 콜 수(×5청크) | 질문생성 층 콜 수 | 예상 wall-clock* |
|---|---|---|---|---|
| 10 (파일럿) | ±0.31 | 50 | 10 | ~1h |
| 30 | ±0.18 | 150 | 30 | ~2.5h |
| 50 | ±0.14 | 250 | 50 | ~4h |
| 100 | ±0.10 | 500 | 100 | ~8h |

\* qwen 청크 콜 90~180s 추정(P03 채점 콜 90.3s + 컨텍스트 가중), workers 4, 보수 페이싱 기준. 정확한 단가는 파일럿에서 실측 후 이 표를 갱신한다.

**권고안**: ① 파일럿 REPEATS=10으로 (a) 콜 단가 실측 (b) 구조 시그니처 정규화가 퇴화하지 않는지(전부 0 또는 전부 1이면 granularity 재조정) 확인 → ② 본실행은 **청크 층 REPEATS=100, 질문생성 층 REPEATS=50**.
- 근거: (1) 청크 층은 대상이 Locked 단일 모델뿐이라 100회여도 총 500콜(P03 20,300콜의 2.5%) — 엄밀성의 한계 비용이 낮고, P03에서 100회가 3회로는 못 잡던 진짜 결함(nemotron 범위 밖 점수)을 잡아낸 실증이 있다. (2) 질문생성 층은 콜당 컨텍스트가 가장 무겁고(그래프 JSON 최대 24k자), 출력 자유도가 높아 완전일치보다 mode_agreement/Jaccard(연속 지표, 더 작은 n에서 안정)가 주 지표가 되므로 50회로 충분. (3) 단, 12시간+ 단일 세션은 D116에서 인프라 붕괴(쿼타 버킷)가 실증된 위험 구간 — 8h 이내로 설계하고 모델 단위가 아닌 **층 단위 순차 실행+raw 병합**으로 부분 실패에 견디게 한다.
- **확정(D125, 2026-07-12)**: 청크 층 REPEATS=100 / 질문생성 층 REPEATS=50 분리안 승인. "최소 100번"은 전역 원칙이 아니라 P03 한정 지시였음을 사용자가 확인 — 파이프라인별 pilot-then-size 산정(위 표)이 원칙으로 유지된다.

## 5. (B) Hook File 재귀 루프 — 도입 전/후 비교 실험 설계

### 5.1 D121 (제안) — Hook File 스키마와 생성기 (`hookfile/` 블록 신설)

- 구성(파일 제안): `hookfile/HOOKFILE_SCHEMA.md`(스키마 문서) · `hookfile/generate_hook_file.py`(생성기) · `hookfile/render_targets.py`(그릇 렌더러) · `hookfile/audit_checklist.py`(체크리스트 자동 감사기) · `hookfile/canary.py`(오염 감시 토큰)
- 스키마 필드(계획 수준 정의 — 각 규칙은 근거 없이는 생성 불가):
  - 파일 레벨: `student_id`, `version`(회차별 증분), `generated_at`, `source_round`, `canary_uuid`(파일마다 고유 — 5.4의 오염 감시용), `coverage`(진단된 취약축 중 처방이 존재하는 비율 — 파일럿 KPI와 직결)
  - 규칙 레벨: `rule_id` · `취약축`(FR-04-01 5축 중 근거 축, ENG-01) · `finding_refs[]`(P02 finding id) · `transcript_refs[]`(P03 인터뷰 턴) · `curriculum_refs{unit, source_pages[]}`(P01 unit_map 기반, CUR-02) · `tech_area`(DASH-15) · `trigger`(적용 조건: 작업 설명과의 유사도 기준 또는 파일 패턴 — Hook 메커니즘의 matcher에 대응) · `지침 본문`(행동 지시 — "기능 구현 전에 endpoint/request/response schema/error case를 표로 먼저 정리" 같은 실행 가능한 절차) · `checkable_condition`(그 지침을 지켰는지 산출물에서 기계 판정 가능한 조건) · `provenance_hash`(근거 측정 산출물의 커밋 해시)
- 생성 방식: **근거 필드는 측정 산출물에서 결정론적으로 조립**하고, LLM(Locked qwen)은 지침 본문의 문장화에만 사용(학생/회차당 1콜 수준) — 처방 내용이 측정 데이터에서 추적 가능해야 "근거 기반 압축"이라는 프레이밍이 코드로 지켜진다.
- 그릇 렌더링 — **확정(D125): 1차 그릇 = 정적 규칙 파일, 단 Claude Code Hooks JSON의 event→matcher→handler 구조를 모티브로 설계**. 즉 CLAUDE.md/AGENTS.md처럼 어디서나 읽히는 정적 문서 포맷을 기본으로 하되, 스키마 자체를 훅 이벤트 구조에 맞춰 짠다(`trigger`=matcher 대응, `checkable_condition`=handler의 판정 로직 대응 — 5.3의 자동 감사기가 이 대응을 실제로 사용). 이렇게 하면 (a) 교육생의 실제 개발 환경(Claude Code 사용 여부)에 의존하지 않고 어디서나 유효하고, (b) 나중에 실제 Claude Code 사용자에게는 같은 스키마를 그대로 실행 가능한 Hooks JSON으로 2차 렌더링할 수 있다 — `render_targets.py`는 정적 파일을 기본 출력, Hooks JSON을 옵션 출력으로 유지.
- 규칙 예산: 초판은 **상위 취약축 3개 × 축당 규칙 최대 3~4개(총 ≤10)** — 과처방은 준수율을 떨어뜨려 효과 측정 자체를 흐린다. 예산 초과분은 우선순위(취약축 심각도 × 재발 횟수) 순으로 절사.
  - WHY: 처방 자산의 가치는 "다 적음"이 아니라 "지킬 수 있음"이다. 준수율(5.3의 지표 3)이 낮으면 효과/무효과를 판정할 수 없게 된다.
  - COST: 절사된 처방은 이번 회차에 전달되지 않는다 — 대신 파일에 `deferred_rules`로 보존해 다음 버전에서 재평가.
  - EXIT: 준수율이 충분히 높게 나오면(예: >0.8) 예산 상향은 상수 하나.

### 5.2 D122 (제안) — 종단(longitudinal) 회차 구조: "측정 중 처방 금지"와 전/후 비교의 양립

**핵심 프레이밍 — 확정 원칙과의 정합**: 운영(제품) 모드에서 Hook File은 수료 시점 1회 산출물(RPT-02)이고 측정 기간 중 주입은 금지다. 실험 모드는 이 원칙을 깨는 것이 아니라 **"측정→수료→처방" 사이클을 k회 반복 인스턴스화**한 것이다 — 각 회차가 하나의 미니 수료 사이클이며, 회차 내부에서는 측정이 처방 없이 완결되고, 처방은 회차 **사이**에만 투입된다. 두 모드가 공유하는 불변식은 하나다:

> **측정기 불변식**: 채점·스캔·인터뷰의 어떤 프롬프트/입력에도 Hook File 내용이 들어가지 않는다. Hook File은 학생(개발 단계)의 환경에만 존재한다. 처방 노출은 "통제된 독립변수"고, 측정기 오염은 "인과를 죽이는 교란"이다 — 전자를 실험하고 후자를 차단한다.

**회차 구조 (k회, 기본 k=4 — 확정(D125))**: 사용자가 k=3에서 k=4로 확정. 근거: 재발률의 "전/후 델타"가 아니라 **회차마다 점진적으로 줄어드는 명확한 곡선**이 산출물이어야 한다는 요구 — 2점(R1/Rk)으로는 우연한 델타와 진짜 감소 추세를 구분할 수 없고, 최소 4점이 있어야 곡선의 형태(단조 감소/포화/재발)를 눈으로도 구분할 수 있다(5.3 지표 1 재정의 참고).

| 단계 | 내용 | Hook File |
|---|---|---|
| R1 개발 | 학생(또는 학생 역할 에이전트)이 과제 T1 수행 | 없음 (baseline) |
| R1 측정 | P02 스캔+판단(콜 0) → P03 인터뷰(turn_engine L1→L3) → FR-04-01 5축 채점 → (선택) P01 unit_map으로 취약 개념의 교안 페이지 매핑 | 측정기는 hook-blind (원래 없음) |

| R2 개발 | 같은 학생이 등가 과제 T2 수행 | **v1 장착** (개발 환경에만) |
| R2 측정 | R1과 동일 절차, 동일 루브릭 | 측정기는 hook-blind (5.4로 강제) |

**비교 설계 3안 (엄밀성 오름차순, 병행 가능)**:

| 설계 | 구조 | 통제하는 교란 | 비용 |
|---|---|---|---|
| **α 자기대조 종단** | 위 표 그대로, 군 1개 | 없음(추세만) — 학습효과와 hook 효과가 섞임 | 최소 |
| **β 평행 대조군** (권장 본실험) | A군(회차마다 hook 누적) vs B군(hook 없이 동일 과제·동일 회차, 수료 시점에만 hook 제공 — 운영 원칙 그대로) → **diff-in-diff**: (A의 R1→Rk 개선) − (B의 R1→Rk 개선) = hook 순효과 | 학습효과·과제 난이도·시점 효과 | 2배 |

- **실행 순서 권고**: γ 먼저(기계 검증 + 효과 크기로 β의 표본 크기 산정) → β는 실제 교육생 파일럿(팀/운영사 일정에 종속, 이 계획 범위 밖 일정). **확정(D125, §7.5 Q8)**: γ 단독 결과도 "β 검증 전" 라벨을 붙여 잠정보고 가능 — 최종/확정 주장만 β 전용으로 남긴다.
- **과제 등가성(Data-First)**: T1~T4 난이도가 다르면 회차 비교가 무효다 — 과제 후보들을 사전에 P02로 스캔해 finding 밀도·언어·규모가 유사한 쌍을 **실측으로** 선별하고, 군 간 과제 순서를 상쇄(counterbalancing)한다. **과제 조달처 확정(D125): 실제 커리큘럼 과제.**
  - WHY: 결과의 생태학적 타당도 — 공개 과제로는 "이 커리큘럼에서 실제로 통하는가"를 답할 수 없다.
  - COST: 과제 풀이 유한해 k=4×2군에 걸쳐 등가 쌍을 선별할 후보군이 좁다 — 회차/코호트 간 과제 재사용 또는 경미한 변형이 필요할 수 있다.
  - EXIT: 등가 쌍이 충분히 안 나오면 공개 과제로 부분 보충(원칙의 명시적 예외로 로그, 수치 조작 없이 결손 명시).
- WHY(설계 β를 본실험으로): "hook 도입 후 좋아졌다"의 최대 대항 가설은 "그냥 두 번째라서 좋아졌다"(연습 효과)다. 평행 대조군 없는 α만으로는 B2B 서사("검증 가능한 결과물")의 '검증'이 성립하지 않는다.
- COST: β는 대조군 학생에게 hook 제공이 수료 시점까지 지연된다 — 운영 원칙(원래 수료 시점 제공)과 동일하므로 윤리·경험 저하 없음. 다만 모집 N이 2배.
- EXIT: β 모집이 불가능하면 α + "targeted/untargeted 내부 대조"(5.3)로 축소 — 인과 강도는 낮아지지만 hook 특이 효과의 부분 증거는 남는다.

- EXIT: 장치가 과하면(개발 마찰) 2·3만 남기고 1·4는 유지 — 1·4는 비용이 사실상 0이라 제거할 이유가 없다.

### 5.5 P01/P02/P03의 hook 루프 적용 지점

- **P02가 루프의 코드 채널 측정기**: 재발 곡선(지표 1)의 코드 채널 산출기이며 콜 0건이라 회차마다 무제한 재실행 가능 — hook 루프의 가장 싼 신호원.
- **P03이 루프의 인터뷰 채널 측정기이자 처방 대상 — 확정(D125, §7.5 Q9)**: "인터뷰에서 드러난 취약점(INTV-06/ENG-01)이 다음 회차 인터뷰 답변 품질을 개선하는가"는 5절 실험과 분리된 별도 실험이 아니라 **같은 종단 루프의 두 번째 채널**로 통합한다 — Hook File 스키마는 이미 `finding_refs`(P02)와 `transcript_refs`(P03) 양쪽을 갖고 있어(5.1) 스키마 변경 없이 가능하다. 효과는 지표 2(FR-04-01 추세)·지표 5(인터뷰 방어 도달률)로 코드 채널과 나란히 곡선화된다 — 두 채널 모두에서 감소/개선 곡선이 보여야 "이점이 명확하다"는 사용자 요구가 충족된다. 측정기 불변식은 두 채널 모두 동일 적용(인터뷰 질문생성·채점은 항상 hook-blind).
- **P01은 처방의 근거 공급기**: hook 규칙의 `curriculum_refs`가 P01 unit_map에서 나온다(CUR-02). 즉 P01 벤치(4절)의 출처 정합률이 낮으면 hook의 교안 근거도 오염된다 — **M2(P01 벤치)가 M4(루프 실험)의 선행 조건**인 이유.
- P01 자체의 "hook 도입 전/후"(교안 분석 품질이 hook으로 좋아지는가)는 이번 범위에서 제외 — P01의 소비자는 학생이 아니라 시스템이라 처방 대상이 다르다. 필요 시 후속 계획으로.
- **규칙 예산(5.1의 ≤10)은 P02+P03 두 채널 합산 상한** — 출처별 배분을 인위적으로 반반 고정하지 않고, 해당 회차에 ENG-01이 지목한 축 심각도로 자연 결정한다(코드 취약점이 심하면 코드 규칙이 더 많이, 반대면 인터뷰 규칙이 더 많이).

## 6. (C) D124 (제안) — 파이프라인 03의 역할: 완료된 템플릿 + 루프의 측정기

- **역할 1 — 방법론 원형(변경 없음)**: 4축 정의·REPEATS 확장 이력(3→100)과 집계 함정 교훈(D115)·인프라 annotation 규약(D109/D111/D116)·키풀/페이싱 실증(D112/D113)·`benchmarks/harness.py`를 P01/P02 벤치가 그대로 상속한다. P03 자체의 재측정은 이번 계획에 없다(1.3의 표가 확정본).
- **역할 2 — hook 루프의 측정기**: 5절 실험에서 turn_engine+채점기는 실험 **도구**지 실험 **대상**이 아니다. 측정기 불변식(5.2)이 적용되는 바로 그 컴포넌트. 채점은 Locked 모델 + 3회 중앙값(재현성 0.273 실측이 강제하는 설계 — 5.3 지표 2).
- **역할 3 — 확정(D125, §7.5 Q9): P03도 hook 루프의 1급 대상**. "인터뷰 결과로 만든 Hook File이 다음 회차 인터뷰의 답변 품질을 개선하는가"를 별도 실험으로 새로 파지 않고 **5절 실험의 인터뷰 채널로 통합**한다(5.5) — 개선 주장의 대상은 항상 **학생의 답변 품질**이지 채점기가 아니며, 측정기(질문생성+채점)는 여전히 hook-blind(측정기 불변식 그대로 적용). 사용자가 이 트랙을 확정한 이유: "이점이 명확하게 보이는" 채널이라서 — 코드 채널(P02)은 정적 스캔이라 변화가 미묘할 수 있는 반면, 인터뷰 채널은 FR-04-01 점수·방어 도달률처럼 사람이 바로 읽을 수 있는 지표라 곡선의 설득력이 크다.
